# 专家辩论：AI协作课程应该怎么教？

## 辩题
面向高职学生的AI协作开发课程，核心重点应该放在哪里？

---

## 第一轮：开场陈词

### 产品经理（正方）

我先说结论：**这门课最大的问题是"产品思维"不够。**

看看现有内容：
- L04-L06 讲需求，只有12学时，还要塞下"发现问题、定义需求、产品设计"三件事
- 没有教竞品分析——学生怎么知道自己的产品是不是在造轮子？
- 没有教用户调研——学生怎么知道用户真的需要这个功能？
- 没有教需求验证——学生怎么知道自己想的方向对不对？

这不是"AI协作"，这是"AI代工"。学生只是学会了对AI下指令，但不知道该下什么指令。

就像教人开车，你只教了"怎么踩油门刹车"，但没教"怎么判断路况、怎么规划路线"。

### AI研究者（反方）

我先指出一个危险信号：**教程里写着"你不需要看懂代码"。**

这句话的潜台词是：让AI帮你做事，你不用理解它在做什么。

后果是什么？
- AI写的代码有bug，学生看不懂，改不了
- AI选了不合适的技术方案，学生不知道
- 遇到AI解决不了的问题，学生束手无策

这培养出来的是"能做但不懂"的人。在职场上，这种人一遇到变故就废了。

我不是说每个人都要成为程序员，但至少要能：
- 看懂AI写了什么
- 判断写得到底对不对
- 知道去哪里查资料

**代码阅读能力，比产品方法论更基础、更紧迫。**

---

## 第二轮：辩论交锋

### 产品经理

我不同意"代码阅读能力更紧迫"。

首先，高职学生毕业后多数不会做纯开发工作。他们更可能做运营、产品助理、项目管理。这些岗位需要的是：发现问题、定义问题、协调资源。

其次，AI会越来越强。现在的AI代码能力已经很强了，两年后可能更强。"看懂代码"这个技能的边际价值在下降。

但"产品思维"永远稀缺。AI能写代码，但AI不能告诉你"用户真正需要什么"。

你教学生看代码，不如教学生：
- 怎么做用户访谈
- 怎么分析竞品
- 怎么验证需求

### AI研究者

你说AI越来越强，那我问你：AI要是出错了怎么办？

举个真实的例子：AI生成了一个登录功能，看起来能用。但实际上：
- 密码是明文存储的
- 没有防暴力破解机制
- session过期时间设置不合理

学生看不懂代码，怎么验收？靠"测试"？测试只能发现"不能用"的问题，发现不了"能用但不安全"的问题。

**你说产品思维稀缺，我同意。但你不能建在沙滩上。**

没有技术理解力的产品思维，就像不懂建筑的建筑师——图画画得再漂亮，楼也是要塌的。

### 产品经理

那我问你：64学时，你打算分多少给学生学代码？

现在开发阶段已经占30学时了。你再加强代码理解，是不是要压缩需求阶段的学时？

别忘了，这些学生不是计算机本科，他们基础本来就弱。你让他学Vue、学Express、学SQL，64学时够吗？

学完出来，可能：
- 产品思维没建立起来
- 代码也只懂皮毛
- 两头都不着

还不如**有所取舍**，把产品思维讲透。

### AI研究者

不是要培养程序员，是要培养"有技术判断力的人"。

我的建议是：
- 不是教学生"怎么写代码"
- 而是教学生"怎么读代码"
- 重点是"读AI写的代码"，理解逻辑，判断对错

这比从头学编程效率高得多。

具体做法：
1. 每次AI生成代码后，增加一个"代码讲解"环节
2. 让AI解释它写了什么，为什么这么写
3. 学生不需要会写，但要能看懂、能提问

**这不是增加负担，而是改变学习方式。**

---

## 第三轮：收尾陈词

### 产品经理

我承认你说的有道理，代码理解力确实重要。

但我想强调一点：**这门课的核心目标是"能独立完成项目"，不是"成为全栈工程师"。**

高职学生的就业现实是：
- 很少有人去做纯开发
- 更多是运营、产品、项目类的岗位
- 技术能力是加分项，不是必选项

如果课程要取舍，我倾向于：
- 砍掉一些技术细节
- 加强产品方法论
- 让学生能"带着想法找AI协作"

**不是每个指挥官都要会开枪，但每个指挥官都要知道仗该怎么打。**

### AI研究者

我同意这门课的目标是"能独立完成项目"。

但什么叫"能独立"？如果：
- 离开AI就不会干活
- AI错了也不知道
- 遇到新问题就傻眼

这叫"能独立"吗？

我的担忧是：这种教学方式可能培养出"AI依赖症"。短期看，学生能做出东西来，很有成就感。长期看，他们在职场上会遇到瓶颈。

**我建议在"用AI"和"懂AI"之间，至少要有30%的时间花在"懂"上。**

不是学成专家，但至少要知道：
- AI能做什么、不能做什么
- 怎么判断AI的输出
- 什么时候该相信AI，什么时候要自己验证

---

## 折中方案

经过辩论，两位专家达成了以下共识：

### 核心分歧的本质

| 产品经理视角 | AI研究者视角 |
|-------------|-------------|
| 职场需要的是"能提出问题的人" | 职场需要的是"能判断方案的人" |
| 产品思维是稀缺能力 | 技术判断力是基础能力 |
| AI会越来越强，代码能力贬值 | AI会犯错，验证能力永远重要 |

**两方都对。问题不是"选哪个"，而是"怎么平衡"。**

### 具体调整建议

#### 1. 不增加总学时，但调整分配

| 调整项 | 原方案 | 新方案 | 理由 |
|--------|--------|--------|------|
| 需求阶段 | 12学时 | 10学时 | 适当压缩 |
| 开发阶段 | 30学时 | 28学时 | 适当压缩 |
| 新增环节 | 0 | 4学时 | 代码理解+验收方法 |
| 调整后总计 | 70学时 | 70学时 | 总量不变 |

#### 2. 新增内容：代码理解模块

在开发阶段的每个功能完成后，增加"代码讲解"环节：

```
原来的流程：
1. 提需求 → AI写代码 → 测试功能 → 下一个需求

新的流程：
1. 提需求 → AI写代码 → 让AI讲解代码 → 学生提问 → 测试功能 → 下一个需求
```

**不是让学生学写代码，而是让学生学会"审问代码"。**

#### 3. 新增内容：产品方法模块

在需求阶段增加两个小节（从12学时压缩到10学时，但增加密度）：

| 原内容 | 新增内容 |
|--------|----------|
| 发现需求（3学时） | 竞品分析（1学时内嵌） |
| 定义需求（4学时） | 用户访谈方法（1学时内嵌） |
| 产品设计（5学时） | 需求验证方法（1学时内嵌） |

#### 4. 教学方法的调整

**原来**：代码看不懂没关系，能用就行

**调整后**：
- 代码看不懂可以，但要会问
- 关键逻辑必须能解释
- 安全相关的代码必须理解

**原来**：产品想清楚再做

**调整后**：
- 产品要先小范围验证
- 不追求完美，追求"验证-改进"的循环

### 最终的平衡点

```
产品思维 ████████████░░░░░░░░  60%
技术理解 ████████░░░░░░░░░░░░  40%
```

不是50:50，因为考虑到高职学生的就业去向，产品思维确实更重要一些。

但40%的技术理解是底线——没有这个底座，上面的东西都是空中楼阁。

### 给教师的建议

1. **不要两种极端**
   - 极端A：只教产品，代码全靠AI
   - 极端B：只教技术，产品草草带过

2. **代码讲解是关键环节**
   - 不要跳过"让AI解释代码"这个步骤
   - 鼓励学生提问"为什么要这么写"
   - 对关键逻辑要做"代码走读"

3. **产品方法要接地气**
   - 竞品分析不用很复杂，但要有
   - 用户访谈不用很专业，但要做
   - 需求验证不用很严谨，但要试

4. **根据学生情况调整**
   - 基础好的班，可以加强技术深度
   - 基础弱的班，侧重产品思维
   - 但两者都不能为零

---

## 辩论总结

这场辩论的核心是：**AI时代，技术能力还需要学吗？学到什么程度？**

答案不是简单的"要"或"不要"，而是：

**就像开车不需要会修车，但需要知道"车出了什么问题该找谁"。**

AI协作也一样：
- 不需要自己写代码
- 但需要能判断代码对不对
- 需要知道AI的边界在哪里
- 需要能在AI帮不上忙时有备选方案

这门课的目标，是培养"能用AI干活，但不被AI取代"的人。
